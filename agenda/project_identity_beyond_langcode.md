# Project identity & provenance: never join two projects by label again

- **Scope & relationships:** azt-collab daemon (registry, share/sync
  keying, merge gates) + client UI (picker, share dialog). Born from
  the 2026-07-22 nml cross-merge (sibling item:
  disentangle_nml_repositories.md). The 0.54.19 no-common-ancestor
  merge guard is the floor; it does NOT cover forked-then-separated
  projects, which share ancestry by nature.
- **Vision / done-criteria:** clicking "share X" can never bind a
  peer to a different project than the user believes X is; two
  projects for the same language coexist on one device and across
  peers without ever cross-merging; every share/sync surface shows
  which directory/remote it is bound to.
- **Deadline:** none (design with Kent before building)
- **Waiting on:** Nothing

## Plans

### Tier 1 — provenance made visible (cheap, ship first)
- Picker buttons + share dialog + peers screen show the binding:
  working_dir tail, remote host (stored spelling), entry/commit
  counts. The share CONFIRM popup states the full binding:
  "Share 'nml' — ~/Assignment/…/WeSay/nml-x-test, no remote —
  with <device>?" (2026-07-22: this alone would have prevented
  the incident).
- Fold into the sync status board item (board rows already need
  the same provenance columns).

### Tier 2 — naming default (Kent's rule, 2026-07-22)
Registration derives the PROJECT NAME as: if the vernlang code is
contained in the LIFT filename or the directory name, default the
name to the LONGEST of the three (vernlang, lift stem, dir name).
The WeSay copy would have auto-named itself `nml-x-test`; the bare
label `nml` stays available for the real project. Corollary:
project NAME (registry/share/sync key, display) and VERNLANG (the
LIFT `lang=` write tag) become distinct fields — LIFT writes keep
using vernlang; wire/share flows key on name. Migration: existing
registrations keep their keys; add `vernlang` field defaulted from
key; peers' `shared_projects` entries are names already.

### Tier 3 — real identity (design discussion needed)
A project id minted at creation, carried in-tree (e.g.
`.azt/project_id`) so clones inherit it; explicitly REGENERATED by
a "fork to separate project" gesture. Share allowlists and merge
gates key on id: same-name-different-id → typed decision, never
auto-merge (closes the forked-projects hole the 0.54.19 guard
can't). Wire-format addition → MIN floors bump when it ships.
