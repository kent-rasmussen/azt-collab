2026-05-04 — clone-auth UX + URL-based credential routing
============================================================================

Problem
-------
User reported that cloning a private GitHub repo from the project picker
showed only "clone failed: not found" with no recovery path. The user
also asked that GitHub vs GitLab credentials be selected automatically
based on the repo URL, since users will pick a LIFT project before ever
visiting the settings page.

Investigation
-------------
- Credentials *are* stored in independent github/gitlab blocks
  (azt_collabd/store.py:4-22), but selection was driven by a single
  `collab_host` setting (store.get_sync_credentials), not by URL host.
- Daemon clone path (server._clone_worker → repo.clone_repo) returned
  CLONE_FAILED with the dulwich error string. The auth-shaped retry only
  matched "credential" / "auth" keywords, missing the 404 case GitHub
  returns for unauthenticated access to private repos.
- Picker UI (azt_collabd/ui/picker_app.py:_after_clone_fail) showed a
  generic _show_error("Clone failed: ...") modal with no jump to
  settings.

2026-05-04 (c) — UI i18n: client owns azt_collab_client.po with fallback
============================================================================

Problem
-------
The picker is the first page users see, but it had no UI language
control and (when run as a subprocess) couldn't translate at all —
the default translator's ``from i18n import _`` only resolves when
launched from the recorder's directory. User asked for live
retranslation with the toggle in **settings**, plus per-module
gettext catalogs (no duplication, "client strings owned in the
client").

Architecture
------------
- New gettext domain ``azt_collab_client`` lives inside the client
  package: ``azt_collab_client/locales/<lang>/LC_MESSAGES/azt_collab_client.po``.
- Recorder keeps its existing ``aztrecorder`` domain. Client catalog
  is its own; no string duplication.
- ``azt_collab_client.translate.tr()`` defaults to the client
  catalog. Hosts call ``set_translator(host_tr)`` to override; the
  override is tried first, then ``tr()`` falls back to the client
  catalog when the host translator returned the msgid unchanged.
  Embedded peers therefore get client strings transparently without
  any recorder-side change required for *this* PR. (A future
  improvement on the recorder would be to chain via gettext's native
  ``add_fallback`` for cleaner semantics, but that's optional.)
- UI language is persisted in ``$AZT_HOME/config.json`` under
  ``ui.language`` so settings and picker subprocesses converge on the
  same language without a coordination channel.
- Live retranslation in *settings* is in-process (clear+rebuild the
  ScreenManager). Live retranslation in the *picker subprocess* is
  driven by mtime polling on ``config.json`` (1 Hz) and the same
  rebuild dance.

Changes
-------

17) NEW azt_collab_client/i18n.py — gettext wrapper + pure-Python
    msgfmt for our domain. Public API:
    set_language / current_language / available_languages /
    language_pref / _ / gettext_translation.

    Auto-applies the persisted language at import time so subprocess
    pickers/settings start in the right language with no setup.
    Auto-compiles ``.po`` → ``.mo`` lazily on first ``set_language``
    whenever the ``.mo`` is missing or older than the ``.po`` — the
    compiler is a ~80-line pure-Python implementation of the GNU MO
    format (single magic, sorted msgid array, two parallel offset
    tables, packed via ``struct``). This keeps peers free of an
    external ``msgfmt`` build step.


18) NEW azt_collab_client/locales/fr/LC_MESSAGES/azt_collab_client.po
    — French translations of every client-owned string: picker UI
    (3), popups (10), langpicker (8), translate.py status messages
    (~50), settings UI (~25), and picker_app.py error / loading
    messages (~6). About 100 entries.

    The matching ``.mo`` is generated lazily by ``i18n._ensure_mo``;
    not committed here (would just round-trip on every PO edit).

Recorder-side follow-up (out of scope here)
-------------------------------------------
The recorder's ``i18n.py`` should chain its own translation with the
client's via gettext's native fallback:

    recorder_t = gettext.translation('aztrecorder', ...)
    client_t   = azt_collab_client.i18n.gettext_translation()
    recorder_t.add_fallback(client_t)

That makes the embedded picker case symmetric with the
``translate.tr`` fallback we already have here. Without it, the
recorder still works (the ``translate.tr`` fallback covers client
strings), but the recorder's own ``_('...')`` outside KV wouldn't
benefit from the client catalog.


Test coverage
-------------
There are no unit tests in this repo by design (per
azt-collab/CLAUDE.md). The recorder-side step scripts at
../azt_recorder/tests/stepN.sh are the canonical verification path. A
manual repro (clone a private GitHub repo from the picker without
authenticating first) should now show the auth-prompt modal with the
"Open settings" button, and after authenticating + retrying the clone
should succeed.
